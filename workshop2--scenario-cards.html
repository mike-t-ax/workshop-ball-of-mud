The Growing Startup

Situation
  • Started 2 years ago with 3 developers
  • Now have 12 developers on the team
  • Everyone works on the same codebase
  • Lots of merge conflicts happening daily
  • Developers constantly stepping on each other's work
The Pain

"I can't work on feature X because someone else is changing the same code areas. We spend more time resolving conflicts than writing new code."

  • Code reviews take forever with 12 people involved
  • Hard to assign clear ownership of different parts
  • Simple changes require coordination across the whole team
Current Tech
  • One big Rails/Django/Node.js application
  • Single shared database
  • Everyone deploys to the same staging environment
  • Shared CI/CD pipeline that gets backed up
Your Challenge
Help them organize their code and team structure so people can work in parallel without constant conflicts and coordination overhead.

The Slow Parts

Situation
  • E-commerce app with 100,000 active users
  • Most features work fine and are reasonably fast
  • BUT: Book search is super slow (5-10 seconds)
  • AND: Recommendation engine kills the database
  • When recommendations run, everything else becomes slow
The Pain

"When someone searches for books, nobody else can load pages properly. The whole site crawls to a halt during peak hours."

  • Server crashes during busy shopping periods
  • Can't scale just the problematic parts
  • Users abandon purchases due to slowness
Current Tech
  • Everything runs in one monolithic application
  • Single database handling all operations
  • Search runs complex SQL queries on main database
  • ML recommendations consume lots of CPU and memory
Your Challenge
Fix the performance problems without making the system too complicated for the team to manage and maintain.

The Reliability Problem

Situation
  • Food delivery app serving thousands daily
  • When ANY part breaks, EVERYTHING goes down
  • Lost orders = very unhappy customers and drivers
  • Small bug in menu loading can break order tracking
The Pain

"If the customer review system crashes, people can't place new orders. A bug in restaurant menu updates brings down the entire app."

  • One bad deployment kills the entire platform
  • Hard to isolate and fix problems quickly
  • Critical features depend on non-critical ones
Current Tech & Critical Features

Current: Monolithic application where everything is connected

Critical Features (MUST always work):

  • üî• Order placement and confirmation
  • üî• Order tracking for customers and drivers
  • üî• Payment processing

Nice-to-have Features: Reviews, promotions, loyalty points

Your Challenge
Make the critical features reliable and always available, even when other parts of the system fail or need maintenance.

The Messy Codebase

Situation
  • 15-year-old application with lots of history
  • Multiple developers have worked on it over time
  • No clear patterns or consistent structure
  • User code, order code, and payment code all mixed together
  • Simple changes often break unexpected things
The Pain

"I needed to change the user login flow, but somehow it affected the order reporting system. No one understands why they're connected."

  • Bug fixes consistently create new bugs elsewhere
  • New features take weeks instead of days
  • Developers are afraid to refactor or clean up code
  • Onboarding new team members takes forever
Current Tech
  • One large application with everything mixed together
  • User authentication code scattered throughout
  • Order processing logic mixed with UI and database code
  • Payment handling embedded in random places
  • Tests are fragile and often break when code changes
  • No clear separation between different business domains
  • Database schema that nobody fully understands
Your Challenge
Clean up the mess and organize the code so changes don't have unexpected side effects. Make it maintainable for the long term without breaking everything in the process.
